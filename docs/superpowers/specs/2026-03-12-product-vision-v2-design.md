# PushUpTracker — Product Vision v2

**Date:** 2026-03-12
**Status:** Approved by user
**Session:** Brainstorming — feature discovery

---

## Контекст

Продукт развивается в двух направлениях:
1. **Внутри компании** — инструмент для корпоративного фитнеса, несколько дисциплин, все сотрудники участвуют где им интересно
2. **Для клиентов** — платформа предоставляется клиентам, у каждого своя изолированная организация

---

## Структура платформы

```
Платформа
├── Публичные комнаты (создаёт платформа)
│   ├── Видны всем зарегистрированным пользователям
│   ├── Если дисциплин 2+ → экран выбора дисциплины
│   └── Внутри: лидерборд + тренировка
│
└── Организации (приватные комнаты для компаний-клиентов)
    ├── Доступ по инвайт-ссылке, полная изоляция
    ├── URL: /org/[slug]
    ├── Если дисциплин 2+ → экран выбора дисциплины
    └── Внутри: лидерборд + тренировка
```

Организация = приватная комната. Оба типа используют одну механику (дисциплины, лидерборд, тренировка).

## Блок 2: Многодисциплинарные комнаты

### Ключевое изменение vs Phase 2

Phase 2 предполагал одну дисциплину на комнату. Теперь комната может содержать несколько дисциплин.

### Модель данных

```prisma
model Room {
  id              String          @id @default(cuid())
  name            String
  code            String          @unique
  slug            String?         @unique   // для организаций: /org/[slug]
  isPublic        Boolean         @default(false)
  isOrg           Boolean         @default(false)
  ownerId         String?
  logoUrl         String?
  inviteToken     String?         @unique   // для организаций
  inviteTokenCreatedAt DateTime?            // для перегенерации
  createdAt       DateTime        @default(now())
  disciplines     RoomDiscipline[]
  participants    Participant[]
}

model RoomDiscipline {
  id         String @id @default(cuid())
  roomId     String
  discipline String  // "pushups" | "squats" | "pullups" | "situps"
  room       Room   @relation(fields: [roomId], references: [id], onDelete: Cascade)

  @@unique([roomId, discipline])
}

model Participant {
  id         String    @id @default(cuid())
  roomId     String
  userId     String?
  discipline String    // к какой дисциплине относится участник
  name       String
  createdAt  DateTime  @default(now())
  room       Room      @relation(fields: [roomId], references: [id], onDelete: Cascade)
  sessions   Session[]

  @@unique([roomId, userId, discipline])  // один userId = одна запись на дисциплину в комнате
}

model Profile {
  id        String   @id
  email     String   @unique
  name      String
  role      String   @default("user")  // "user" | "superadmin"
  createdAt DateTime @default(now())
}

// Org-admin роль: хранится отдельно от Profile.role (который только для superadmin)
// Детальная схема определяется в Phase 5 плане
model OrgAdmin {
  id        String   @id @default(cuid())
  roomId    String   // организация (Room с isOrg=true)
  userId    String   // Profile.id
  createdAt DateTime @default(now())

  @@unique([roomId, userId])
}

model Session {
  id            String      @id @default(cuid())
  participantId String
  count         Int
  duration      Float
  date          String      // формат: "YYYY-MM-DD" (ISO date)
  createdAt     DateTime    @default(now())
  participant   Participant @relation(fields: [participantId], references: [id], onDelete: Cascade)
}
```

### Дисциплины (фиксированный список)

`pushups` · `squats` · `pullups` · `situps`

Расширяется по мере добавления MediaPipe-трекеров.

### UX выбора дисциплины

- Если в комнате **одна дисциплина** → сразу попадаешь внутрь (без экрана выбора)
- Если **2+ дисциплин** → экран выбора с карточками дисциплин
- Участник привязан к конкретной дисциплине (`Participant.discipline`)
- Один пользователь может участвовать в **нескольких дисциплинах** одной комнаты — каждая дисциплина создаёт отдельную `Participant` запись

### Удаление дисциплины из комнаты

При удалении дисциплины из `RoomDiscipline`:
- `Participant` записи этой дисциплины остаются (данные не теряются)
- Они становятся архивными — не отображаются в активном лидерборде
- Org-admin видит предупреждение перед удалением: "X участников потеряют доступ к этой дисциплине"

### Что меняется в Phase 2 плане

- `Room.discipline String` → удаляется, заменяется на `RoomDiscipline[]`
- Join flow: при входе в комнату с 2+ дисциплинами — экран выбора перед join
- `Participant` получает поля `discipline` и уникальный ключ `[roomId, userId, discipline]`
- API `/api/rooms/[code]/join` принимает `{ discipline }`
- **Phase 2 план (`phase2-groups-v2.md`) устарел и требует полного пересмотра**

## Блок 3: Лидерборд по дисциплине

### Что показывает

Все участники выбранной дисциплины в данной комнате/организации, отсортированные по сумме повторений за выбранный период.

### Временные периоды

Переключалка с тремя вариантами:

| Период | Логика фильтрации |
|--------|------------------|
| Неделя | `Session.date >= начало текущей недели (пн)` |
| Месяц | `Session.date >= '2026-03-01'` |
| Всё время | Все сессии |

`Session.date` хранится как `YYYY-MM-DD` строка — подходит для строкового сравнения в PostgreSQL.

### UI

- Текущий пользователь подсвечен `[you]`
- Место (#1, #2, ...) + имя + сумма повторений
- Переключатель: `[неделя] [месяц] [всё время]`

## Блок 4: Организации

### Концепция

Организация = `Room` с `isOrg = true` и уникальным `slug`. Никакой отдельной иерархии нет — дисциплины внутри орга это `RoomDiscipline[]`, участники — `Participant[]` с привязкой к дисциплине.

"Комнаты внутри орга" из Org-Admin панели = возможность org-admin создавать дополнительные приватные Room записи, связанные с оргом через ownerId. Это опциональная фича — на текущем этапе орг = одна Room с несколькими дисциплинами.

### Доступ

- Инвайт-ссылка: `https://platform.com/org/[slug]?invite=[token]`
- `inviteToken` генерируется при создании орга, хранится в `Room.inviteToken`
- Перегенерация инвалидирует старый токен (обновляет поле)
- Регистрация по инвайту → автоматически создаётся `Participant` в орге

### URL

- Сейчас: `/org/[slug]` (работает на любом домене)
- Субдомены — вне скоупа текущих фаз, рассматривается отдельно после смены домена

### Брендинг

- Название орга в хедере
- Логотип в бакете `room-logos` (тот же бакет что и для обычных комнат), файл: `org-{slug}.{ext}`

### Участники

- Присоединяются только по инвайт-ссылке
- Могут быть удалены из орга (например, при увольнении) — их `Participant` записи удаляются cascade
- Пользователь может одновременно состоять в организации И участвовать в публичных комнатах

### Изоляция

- Участники одной организации не видят другие организации
- Лидерборды, участники — полностью изолированы

## Блок 5: Org-Admin панель

### Роль

Администратор организации — пользователь с расширенными правами внутри своего орга.

### Правила

- Минимум 2 админа на организацию
- Проверка при снятии прав (`DELETE /org/[slug]/admins/[userId]`): если останется < 2 админов — API возвращает 400 с ошибкой "Нельзя удалить последнего администратора"
- Первый админ назначается платформой при создании орга

### Возможности

| Действие | Описание |
|----------|----------|
| Управление участниками | Список, удаление из орга |
| Инвайт | Скопировать / перегенерировать инвайт-ссылку |
| Управление дисциплинами | Добавить дисциплину / удалить с предупреждением |
| Управление админами | Назначить участника админом / снять права (с проверкой минимума) |

### Аналитика

Только управление — без аналитики на текущем этапе.

## Блок 6: Super-Admin панель

### Роль

Внутренний инструмент команды платформы. `Profile.role = "superadmin"`.

### Доступ

Роут `/admin/*` защищён middleware: проверяет `Profile.role === "superadmin"`. Первый superadmin назначается вручную через SQL при деплое.

### Возможности

| Действие | Описание |
|----------|----------|
| Создать организацию | Slug, название, логотип, email первого admin |
| Инвайт-ссылка | Сгенерировать / перегенерировать |
| Список организаций | Название, slug, кол-во участников, дата создания |
| Создать публичную комнату | Название, дисциплины, логотип |
| Управление публичными комнатами | Редактировать, удалить |

```
    ↓

    ↓

```
